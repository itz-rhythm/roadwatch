                        ROADWATCH - SUBMISSION DETAILS

1. PROJECT OVERVIEW
--------------------------------------------------------------------------------
RoadWatch is a web-based public accountability and monitoring platform designed 
to track public road infrastructure health. The application aims to solve the 
issue of delayed road repairs, lack of contractor transparency, and bureaucratic 
delays in municipal road maintenance. 

Through RoadWatch, citizens can file geo-located complaints about potholes, 
waterlogging, and other hazards, while municipal authorities can assign tasks 
directly to registered contractors. The platform uses a data-driven approach, 
ranking contractors based on their historic project failure frequency and citizen 
feedback, and utilizes machine learning for severity classification, duplicate 
detection, and predictive maintenance scheduling.


2. SYSTEM ARCHITECTURE & COMPONENTS
--------------------------------------------------------------------------------
The project is built as a multi-service decoupled application:

- Frontend: A responsive Single Page Application (SPA) built using React and 
  Vite. It features interactive spatial map visualization using Leaflet for 
  heatmap analysis, Tailwind CSS for premium responsive UI styling, and Recharts 
  for municipal analytics dashboards.
  
- Backend API: A Node.js and Express server that handles core business logic, 
  user authentication, route planning, real-time updates via Socket.io, PDF 
  generation (for Right to Information documents), and cron jobs for auto-escalating 
  SLA breaches.

- ML Service: A FastAPI (Python) service that handles computationally heavy 
  tasks such as comparing CLIP image embeddings for duplicate check requests, 
  simulated Random Forest inference for road failure predictions, and YOLOv8 
  video frame analysis for automated dashcam issue drafting.

- Database: PostgreSQL with the PostGIS extension, allowing advanced geospatial 
  queries (e.g., finding nearby complaints, calculating road damage length, 
  and checking ward boundaries).

4. KEY ASSUMPTIONS & DESIGN DECISIONS
--------------------------------------------------------------------------------

- In-Memory Database Fallback:
  To make local testing and demonstrations friction-free, the backend is 
  designed with a PostgreSQL database detection check. If a connection to 
  Postgres fails (e.g., if database setup is pending), the backend automatically 
  falls back to an in-memory database mock store (`MEMORY_DB`). This ensures the 
  app is functional immediately upon running locally.

- User Roles & Access Control:
  The system assumes five distinct user roles with specific privileges:
  * Citizen: Can report issues, upvote active complaints, and download RTIs.
  * Volunteer: Can verify nearby citizen complaints to reduce spam.
  * Ward Authority: Can view dashboards, assign contractors, and flag delays.
  * City Engineer: Can review budgets and contractor stats at the city level.
  * Admin: Full system control, including blacklist overrides.

- SLA & Auto-Escalation Logic:
  Service Level Agreements (SLAs) are automatically calculated during complaint 
  submission based on severity:
  * Critical: 24-hour resolution deadline.
  * Dangerous: 48-hour resolution deadline.
  * Moderate: 7-day resolution deadline.
  * Minor: 30-day resolution deadline.
  If a contractor misses this deadline, the backend cron job automatically 
  initiates an escalation flow, increasing the priority level and notifying the 
  zone/city engineer.

- Image Duplicate Detection (CLIP):
  To prevent citizens from submitting duplicate reports for the same pothole or 
  road failure, the platform assumes that issues within a 100-meter radius 
  require a visual similarity check. A CLIP model generates 512-dimension vector 
  embeddings of the photos, and a Cosine Similarity score above 0.85 indicates a 
  duplicate submission.

- Mock vs. Production ML Inference:
  For lightweight local deployment, complex operations (such as loading the 
  YOLOv8 weight files for dashcam parsing or running heavy neural network feeds) 
  are stubbed with structured mock responses. This mirrors production API 
  payload formats without requiring high-end GPU hardware to run the project locally.
